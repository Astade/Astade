Mission

This project shall develop an application, which can convert a trace file in text format to sequence diagrams. The application can also modify the diagrams and save in the same format. 

The tool will be needed, for the Project Astade, it should be useful for Programmers of other projects, too.

It shall be written with the Astade tool! Try it, you can download here: http://trace2uml.tigris.org/servlets/ProjectDocumentList?folderID=6208

Some screen shots

The features

  • reads special trace files which can easily be generated by any application.
  • filters this files for a groups of objects.
  • filters this file for a certain time period.
  • draws nice (graphical) sequence diagrams.
  • can modify the diagrams
  • the source (trace file) is pretty good usable as debug trace, too!

The Trace format (example)

900000    *        (!) Object1
900000    Object1  (!)  Object2
900001    Object1  ==> Object2     DoSomething(void*)
900002    Object1  <== Object2     6
900003    Object1  >>> BusyState
900004    Object2  >-- Object1     StartAgain(56)
900005    Object2  ==> Object2     Wait()

# blank lines are ignored 

900005    Object2  INF "Starting to wait"
900006    Object1  --> Object2     StartAgain(56)
900007    Object1  >-- Object2     OK
900008    Object2  --> Object1     OK
900009    Object2  <== Object2    

# Object2 is going to delete itself!
900010    Object2  (X) Object2    

The Trace format (details)

Symbol description

#

 Lines starting with '#' are treated as comment.
(!) this is used to notify an Object creation.
-X- this is used to notify an Object delete.
>-- this is used to notify an asynchronous message send.
--> this is used to notify an asynchronous message receive.
==> this is used to notify a synchronous function call.
<== this is used to notify a synchronous function return.
>>> this is used to notify a state change. Only useful when reading the Trace for debugging purpose, no influence for the diagrams.
INF this is used to notify a general debugging info. Only useful when reading the Trace for debugging purpose, no influence for the diagrams.

ERR

 this is used to notify an error. Only useful when reading the Trace for debugging purpose, no influence for the diagrams.


Remarks:

  • Returns are always in the opposite order than calls! Therefore the first (calling) object name in the trace is redundant and may be "*"
  • Messages receive always in the order they are send. Therefore the second (sender) object name and the message label can be "*"
  • Timestamps might be omitted.
  • "void" may be omitted in returns.

Redundant information

As stated before, there is redundant information in the trace format, which is only for better readability. This is:

  • the "sender" in received messages (because its already known from the "send" entry)
  • the "label" of a received message (because its already known from the "send" entry)
  • the "receiver" of a return (because its already known from the previous call)

The program ignores this values and replace them with the value of the corresponding "send" call. That means, the can be omitted and its not recognized, when they differ in the trace.
